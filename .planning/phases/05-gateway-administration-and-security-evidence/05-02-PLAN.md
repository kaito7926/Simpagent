## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> FastAPI OAuth routes | Untrusted browser traffic starts and completes the provider redirect flow. |
| FastAPI -> Google OAuth endpoints | Provider token exchange uses environment-held secrets and returns external identity claims. |
| OAuth provider identity -> local account/session model | External identity proof is linked to internal user accounts and session issuance. |
| readiness API -> downstream auth shell | Backend capability truth feeds later frontend CTA rendering and must not expose secrets. |
| human approval -> dependency manifest | The OAuth dependency can enter the repo only after explicit provenance review. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02 | Spoofing | `backend/app/api/routes/auth_oauth.py` | mitigate | Validate OAuth `state` on callback per D-04, keep redirect/callback fully backend-owned, and deny callback reuse or missing state. |
| T-05-03 | Elevation of Privilege | `backend/app/identity/oauth_service.py` | mitigate | Reuse existing first-party session issuance and persist identity by `(issuer, subject)` per D-05 and D-06 instead of trusting mutable email as the durable key. |
| T-05-04 | Information Disclosure | readiness and OAuth error paths | mitigate | Expose only configured/unconfigured provider capability state; never surface client secrets or provider payloads in readiness, errors, or browser-visible redirects. |
| T-05-SC | Tampering | Python package install path | mitigate | Hold the blocking Authlib provenance checkpoint inside this plan and pin Authlib only after explicit human approval from the package-legitimacy audit. |
</threat_model>
