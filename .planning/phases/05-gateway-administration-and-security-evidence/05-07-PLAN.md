<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> admin API wrappers | Untrusted client UI requests sensitive admin evidence through protected endpoints. |
| backend evidence payload -> drawer UI | Sanitized snippets cross from backend contracts into browser detail views. |
| admin route exposure -> shared shell | Newly exposed backend gateway-evidence routes must preserve the same backend authority as existing admin surfaces. |
| admin navigation -> protected surfaces | UI must not imply access when backend role/scope checks deny it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-22 | Information Disclosure | `frontend/components/admin/EvidenceDetailDrawer.tsx` | mitigate | Render only backend-sanitized snippet fields and bounded detail sections; do not add raw JSON or payload dumps per D-20. |
| T-05-23 | Elevation of Privilege | shared admin shell and `backend/app/api/routes/admin.py` | mitigate | Keep all meaningful authorization in backend API calls, expose gateway evidence only through `admin:read`-guarded routes, reflect denied states explicitly, and preserve `admin:read` / `admin:write` distinctions per D-17 and AUTHZ-02. |
| T-05-24 | Spoofing | admin navigation affordances | mitigate | Use backend-provided data and scopes for page content; never fabricate admin evidence or pretend a denied user has access. |
| T-05-25 | Repudiation | admin detail inspection | mitigate | Preserve correlation IDs and row-level references in evidence tables/drawers so viewed evidence remains traceable. |
</threat_model>
