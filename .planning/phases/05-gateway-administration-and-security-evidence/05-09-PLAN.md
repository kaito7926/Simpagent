<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| readiness API -> auth shell | Backend capability truth controls whether each OAuth CTA is shown or actionable. |
| browser -> backend OAuth start routes | Client navigation begins the provider redirect flow without exposing secrets to the browser. |
| auth shell -> local auth form | New provider CTAs must not break the existing protected local login/register flow. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-30 | Information Disclosure | `frontend/lib/auth-session.ts` | mitigate | Start OAuth only by browser navigation to backend routes and never persist access/refresh tokens in browser storage per D-05. |
| T-05-31 | Spoofing | provider CTA availability | mitigate | Drive Google/GitHub button state strictly from backend readiness truth per D-10 instead of hardcoded UI assumptions. |
| T-05-32 | Tampering | auth-shell placeholder controls | mitigate | Remove fake disabled social controls and replace them with factual enabled/disabled states per D-26 and D-28. |
| T-05-33 | Denial of Service | auth-shell login surface | mitigate | Preserve local login/register availability in all provider readiness states so missing OAuth config does not block account access. |
</threat_model>
