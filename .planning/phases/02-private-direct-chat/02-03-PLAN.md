<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Chat route -> provider adapter | Owner-checked conversation content leaves the app for the configured LLM provider |
| FastAPI -> PostgreSQL | Turn state, idempotency keys, and sequence numbers become durable |
| Provider -> FastAPI | Untrusted provider response or failure returns to application state |
| Principal claims -> chat mutation path | Token scopes and current DB user state decide whether provider work may start |

## STRIDE Threat Register

| Threat ID | Category | Severity | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|---|
| T-02-09 | Tampering | HIGH - blocking | message send state machine | mitigate | client_message_id uniqueness, owner row lock, pending-turn conflict, monotonic sequence_no tests |
| T-02-10 | Information Disclosure | HIGH - blocking | provider request context | mitigate | Build provider payload only from owner-checked local history; no tokens, cookies, secrets, or cross-user messages |
| T-02-11 | Repudiation | HIGH | provider failure handling | mitigate | Persist failed assistant state with correlation ID and retryable metadata |
| T-02-12 | Spoofing | HIGH - blocking | provider success state | mitigate | Treat empty/status/error provider responses as ChatProviderError; never fabricate assistant content |
| T-02-13 | Denial of Service | MEDIUM | provider latency and retries | mitigate | Bounded adapter timeout, low SDK retry ceiling, one in-flight turn per conversation |
| T-02-SC | Tampering | MEDIUM | package supply chain | transfer | openai package legitimacy is handled by blocking gate in 02-02 before this plan executes |
</threat_model>
