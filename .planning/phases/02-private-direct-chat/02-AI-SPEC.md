## 1. System Classification

**System Type:** Conversational

**Description:**
Authenticated users hold durable, owner-only direct conversations with an OpenAI-compatible LLM through a safe browser workspace. A good Phase 2 response path accepts one user message, persists it once, sends bounded non-streaming context to the configured provider, stores exactly one successful assistant response, surfaces retryable provider failures without fabricating content, and renders Markdown/code safely without crossing user ownership, scope, or browser-execution boundaries.

**Critical Failure Modes:**
1. Cross-user access or inference of another user's conversations or messages.
2. Duplicate submit/retry behavior creates duplicate provider calls, duplicate user messages, or out-of-order assistant messages.
3. Provider failure is hidden by a fabricated successful assistant response or an error without a stable correlation ID.
4. User/model Markdown executes raw HTML, script handlers, or dangerous URLs in the browser.
5. Secrets, bearer tokens, cookies, provider keys, or sensitive raw content are logged, sent to the model unnecessarily, or exposed in errors.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Secure SaaS / small-company service chatbot, delivered as a university security prototype.

**User Population:** Authenticated small-company users and project evaluators using a browser chat workspace for private direct LLM conversations. Phase 2 users are not using web search, Python tools, RAG, shared workspaces, or regulated professional workflows.

**Stakes Level:** High

**Output Consequence:** The assistant response is persisted to the user's private conversation history and rendered in the browser. When the system gets privacy, ownership, retry, failure, or rendering behavior wrong, the consequence is private-chat disclosure, duplicate provider work/cost, misleading success state, broken user trust, or browser compromise through untrusted Markdown.

### What Domain Experts Evaluate Against

Dimension: Private conversation boundary
Good (domain expert would accept): Every list, retrieve, delete, send, retry, and history-context path proves ownership in the same data access operation, and a second authenticated user receives only a denied/not-found response with no title, message count, timing clue, provider call, or audit content from the target conversation.
Bad (domain expert would flag): The API checks `chat:read` or `chat:write` but then fetches by `conversation_id` alone, lets another user append to a thread, leaks whether a conversation exists, or includes another user's turn in model context.
Stakes: Critical
Source: OWASP API1:2023 Broken Object Level Authorization; project requirements `AUTHZ-05`, `AUTHZ-06`, `TEST-02`.

Dimension: Chat data minimization
Good (domain expert would accept): Provider requests contain only the stable direct-chat instruction and owner-checked conversation turns needed for the current answer; logs, errors, telemetry, and metadata keep correlation IDs, status, latency, model, token counts, and sanitized error codes without bearer tokens, cookies, provider keys, full prompts, or full assistant content.
Bad (domain expert would flag): Request/response logs contain raw chat text or secrets, hidden auth metadata is sent to the model, old unrelated turns are included "just in case", or provider exceptions expose credential-bearing request details.
Stakes: Critical
Source: OWASP LLM06 Sensitive Information Disclosure; NIST AI RMF Generative AI Profile privacy/security risk framing; GDPR Article 25 privacy-by-design principle when personal data is in scope.

Dimension: Idempotent send and retry behavior
Good (domain expert would accept): Double-clicks, browser retries, refreshes, and explicit retry reuse the same `client_message_id` and already-persisted user message; the backend permits only one in-flight assistant turn per conversation and stores exactly one successful assistant response in sequence order.
Bad (domain expert would flag): A retry creates a second user message, starts a second provider call for the same turn, produces assistant responses out of order, or converts a provider failure into a fabricated successful answer.
Stakes: High
Source: SaaS reliability practice for idempotent mutation APIs; project decisions `D-11` through `D-16`; requirements `CHAT-05`, `CHAT-06`, `CHAT-07`.

Dimension: Understandable failure state
Good (domain expert would accept): Provider timeout, rate limit, auth/configuration, empty response, and server failures render as an inline failed assistant row with stable retryability, a user-safe message, and a support correlation ID; no raw stack trace, provider key, prompt body, or misleading assistant text is shown.
Bad (domain expert would flag): The UI silently drops the turn, leaves the user unsure whether the message was sent, shows a generic failure without a correlation ID, exposes provider internals, or displays a confident assistant response after the provider failed.
Stakes: High
Source: NIST AI RMF emphasis on measuring/managing AI system risk and user trust; project requirements `CHAT-07`, `OBS-01`, `OBS-02`.

Dimension: Safe Markdown and code display
Good (domain expert would accept): User and model Markdown render with raw HTML disabled, code blocks displayed as inert text, safe link schemes only, external-link isolation, and negative XSS tests for scripts, event handlers, `javascript:` URLs, and dangerous `data:` URLs.
Bad (domain expert would flag): Rendered chat content can execute script, attach event handlers, navigate through unsafe URL schemes, mutate sanitized HTML after sanitization, or run code from a code block.
Stakes: Critical
Source: OWASP LLM02 Insecure Output Handling; OWASP Cross-Site Scripting Prevention Cheat Sheet; project requirement `CHAT-10`.

### Known Failure Modes in This Domain

- Cross-user conversation exposure: an authenticated user changes a conversation or message ID and can infer, read, modify, delete, or append to another user's private chat.
- Cross-session context leakage: cached context, shared memory, provider-state reuse, or incorrectly scoped history causes one user's prior messages to appear in another user's prompt or answer.
- Retry amplification: frontend resubmission, SDK retry, and backend retry combine into duplicate provider calls, duplicate persisted turns, higher cost, and inconsistent sequence ordering.
- Stored/rendered chat XSS: attacker-supplied Markdown or model output persists in history and later executes in the browser through raw HTML, event handlers, or unsafe links.

### Regulatory / Compliance Context

No sector-specific regime such as HIPAA, GLBA, FCA, or legal professional rules is identified for Phase 2 because this is a general-purpose university prototype, not a healthcare, financial, legal, or employment decision system.

Relevant general constraints still apply to credible evaluation:
- Privacy-by-design and data minimization are the right baseline when conversation content may include personal data, especially if the prototype is adapted for EU users under GDPR Article 25.
- If adapted into a real small-company SaaS serving California consumers, CCPA/CPRA-style access, deletion, disclosure, and reasonable-security expectations would affect retention, deletion, provider-data-flow documentation, and audit evidence.
- If deployed to EU users as a chatbot, EU AI Act transparency expectations mean users should understand they are interacting with an AI system. Phase 2 already satisfies the obvious-chatbot baseline, but documentation should avoid implying human or regulated-professional accountability.
- External LLM provider use is a data-flow and processor/subprocessor concern. Evaluators should verify that prompts sent to providers are limited, configurable, documented, and never include secrets or other users' data.

## 2. Framework Decision

**Selected Framework:** OpenAI Python SDK behind a thin local chat adapter

**Version:** `openai>=2,<3`

**Rationale:**
Phase 2 is a direct, non-streaming conversational chat slice, not RAG, multi-agent orchestration, tool execution, or autonomous planning. The project requirements call for an OpenAI-compatible provider adapter configured by `LLM_API_BASE`, `LLM_API_KEY`, `LLM_MODEL`, timeout, and retry settings, while the project stack explicitly avoids LangChain/LiteLLM in v1. A thin local adapter around the official OpenAI SDK gives the required provider compatibility with the smallest behavioral surface and keeps authorization, idempotency, persistence, observability, and rendering safety in application code where they can be tested directly.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| OpenAI Agents SDK | Adds agent abstractions, handoffs, and guardrails that Phase 2 does not need; it is better reserved for later if provider-native agent features become a conscious product requirement. |
| Google ADK | Required later for Gemini Google Search, but Phase 2 has no Google Search/tool orchestration and should not mix future agent concerns into direct chat. |
| LangChain / LiteLLM | Adds abstraction and transitive dependency surface that the project explicitly excludes from v1 for this direct two-provider design. |

**Vendor Lock-In Accepted:** Partial — the app uses the OpenAI SDK API surface, but routes it through configuration-driven OpenAI-compatible credentials and a local adapter boundary so provider replacement does not spread through route, persistence, or UI code.

### Entry Point Pattern
```python
import asyncio
import os

from openai import AsyncOpenAI


SYSTEM_PROMPT = (
    "You are SimpAgent, a private direct chat assistant. "
    "Answer the user's latest message clearly. Do not claim access to tools, "
    "web search, files, or other users' conversations."
)


async def main() -> None:
    client = AsyncOpenAI(
        api_key=os.environ["LLM_API_KEY"],
        base_url=os.environ.get("LLM_API_BASE") or None,
        timeout=float(os.environ.get("LLM_TIMEOUT_SECONDS", "30")),
        max_retries=int(os.environ.get("LLM_MAX_RETRIES", "1")),
    )

    completion = await client.chat.completions.create(
        model=os.environ.get("LLM_MODEL", "gpt-4o"),
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": "Write a short Python list comprehension example."},
        ],
        temperature=0.3,
        max_completion_tokens=800,
        stream=False,
    )

    print(completion.choices[0].message.content or "")


if __name__ == "__main__":
    asyncio.run(main())
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `AsyncOpenAI` client | Async SDK client backed by HTTPX; accepts `api_key`, `base_url`, `timeout`, and `max_retries`. | Use one injected application-level client for FastAPI request handlers instead of constructing clients inside every route. |
| Chat Completions request | `client.chat.completions.create(model=..., messages=[...])` returns a typed completion with assistant content at `choices[0].message.content`. | Primary Phase 2 endpoint because it is the most broadly supported OpenAI-compatible API shape. |
| Message roles | `system` or OpenAI `developer`, `user`, and `assistant` messages form the bounded conversation context. | Put stable product behavior in the first instruction message, then append owner-checked persisted turns in sequence order. Prefer `system` for broad OpenAI-compatible providers unless the selected provider documents `developer`. |
| SDK error classes | `APIConnectionError`, `APIStatusError`, `RateLimitError`, and related subclasses distinguish network, provider, rate, auth, and server failures. | Convert provider failures into retryable inline chat errors with correlation IDs; never persist fabricated assistant text. |
| Pydantic parse helper | `client.chat.completions.parse(..., response_format=MyModel)` validates structured responses into Pydantic models. | Use only for internal structured sub-results such as safety/metadata checks; normal Phase 2 assistant prose should remain plain text. |

### Common Pitfalls
1. **Assuming every OpenAI-compatible provider supports the Responses API.** The OpenAI SDK documents Responses as the primary API, but many compatible providers still implement only `/v1/chat/completions`. Phase 2 should use Chat Completions by default and hide any later Responses support behind the local adapter.
2. **Letting SDK retries bypass application idempotency.** The SDK retries selected connection, timeout, 408, 409, 429, and 5xx errors by default. Keep `max_retries` low, persist the user message before the provider call, and key retries by `client_message_id` so provider retries never create duplicate messages or duplicate assistant rows.
3. **Logging provider exceptions too verbosely.** `APIStatusError` can expose request/response details. Log provider status, provider request ID, local correlation ID, model name, and retry count, but never log API keys, bearer tokens, cookies, full prompts, or full assistant content.
4. **Mixing provider-stored conversation state with local ownership rules.** Do not depend on `previous_response_id`, hosted conversations, or provider-side `store=true` for Phase 2. The application database is the source of truth for ownership, ordering, deletion, retry, and audit behavior.
5. **Using structured-output helpers for free-form chat.** `chat.completions.parse()` is useful for schema-checked metadata, but it can raise on length/content-filter finish reasons and constrains model output. For normal user-visible chat, request plain text and validate only app-level limits.

## 4. Implementation Guidance

**Model Configuration:**

Use configuration, not hardcoded model names:

| Setting | Phase 2 Value |
|---------|---------------|
| `LLM_API_BASE` | OpenAI-compatible base URL; omit or set to OpenAI default for first-party OpenAI. |
| `LLM_API_KEY` | Required secret loaded from environment or Compose secret; never persisted or logged. |
| `LLM_MODEL` | Default `gpt-4o` for broad Chat Completions compatibility, or the provider-approved equivalent. Validate at startup with a tiny non-user prompt. |
| `LLM_TIMEOUT_SECONDS` | `30` seconds for prototype direct chat. |
| `LLM_MAX_RETRIES` | `1` SDK retry; app-level retry is user-triggered and idempotency-keyed. |
| `temperature` | `0.3` for stable helpful chat. |
| `max_completion_tokens` | `800` initially; raise only with UI and cost-budget evidence. If a compatible provider only supports legacy `max_tokens`, map that inside provider-specific adapter config after an integration test proves it. |
| `stream` | `False` for Phase 2. Streaming is deferred. |

**Core Pattern:**

```python
from dataclasses import dataclass

import openai
from openai import AsyncOpenAI


@dataclass(frozen=True)
class ChatTurn:
    role: str
    content: str


@dataclass(frozen=True)
class ChatCompletionResult:
    content: str
    provider_request_id: str | None
    prompt_tokens: int | None
    completion_tokens: int | None


class ChatProviderError(RuntimeError):
    def __init__(self, code: str, *, retryable: bool, provider_request_id: str | None = None) -> None:
        super().__init__(code)
        self.code = code
        self.retryable = retryable
        self.provider_request_id = provider_request_id


class OpenAIChatAdapter:
    def __init__(
        self,
        *,
        api_key: str,
        base_url: str | None,
        model: str,
        timeout_seconds: float,
        max_retries: int,
    ) -> None:
        self._model = model
        self._client = AsyncOpenAI(
            api_key=api_key,
            base_url=base_url,
            timeout=timeout_seconds,
            max_retries=max_retries,
        )

    async def complete(self, *, turns: list[ChatTurn]) -> ChatCompletionResult:
        messages = [{"role": turn.role, "content": turn.content} for turn in turns]

        try:
            completion = await self._client.chat.completions.create(
                model=self._model,
                messages=messages,
                temperature=0.3,
                max_completion_tokens=800,
                stream=False,
            )
        except openai.RateLimitError as exc:
            raise ChatProviderError(
                "provider_rate_limited",
                retryable=True,
                provider_request_id=getattr(exc, "request_id", None),
            ) from exc
        except openai.APIConnectionError as exc:
            raise ChatProviderError("provider_unreachable", retryable=True) from exc
        except openai.APIStatusError as exc:
            raise ChatProviderError(
                "provider_status_error",
                retryable=exc.status_code >= 500,
                provider_request_id=getattr(exc, "request_id", None),
            ) from exc

        choice = completion.choices[0]
        content = choice.message.content or ""
        if not content.strip():
            raise ChatProviderError("provider_empty_response", retryable=True)

        usage = completion.usage
        return ChatCompletionResult(
            content=content,
            provider_request_id=getattr(completion, "_request_id", None),
            prompt_tokens=getattr(usage, "prompt_tokens", None) if usage else None,
            completion_tokens=getattr(usage, "completion_tokens", None) if usage else None,
        )
```

Route-level usage:

1. Authenticate and resolve principal.
2. Enforce `chat:write` and conversation ownership in the same query path.
3. Acquire a per-conversation in-flight lock or transaction guard.
4. Insert or reuse the user message by `client_message_id`.
5. Build bounded context from local persisted messages only.
6. Call `OpenAIChatAdapter.complete(...)`.
7. Persist exactly one assistant message on success, or persist/update a failed assistant placeholder on provider failure.

**Tool Use:**

No tools are enabled in Phase 2. Do not pass `tools`, `tool_choice`, web-search parameters, file-search parameters, code-interpreter parameters, or MCP configuration to the provider. The system prompt must explicitly avoid claiming tool, file, browser, web, tenant, or account access.

**State Management:**

The database owns all conversation state. Store conversations and messages locally with `user_id`, `conversation_id`, `sequence_no`, `role`, `content`, `status`, `client_message_id`, provider metadata, and timestamps. Use database constraints plus transaction locking to ensure one in-flight assistant turn per conversation and monotonic `sequence_no` assignment.

Do not store raw provider API keys, access tokens, cookies, or full prompt payloads in message metadata. Metadata may include provider name, model, token counts, local correlation ID, provider request ID, retry count, finish reason, latency, and sanitized error code.

**Context Window Strategy:**

For each request, construct context in this order:

1. Stable developer/system prompt.
2. Optional local conversation summary once the thread is long enough.
3. Most recent owner-checked messages in ascending `sequence_no`.
4. Latest user message.

Use a conservative token budget before calling the SDK: reserve at least `max_completion_tokens + 500` tokens for output and safety margin, then trim oldest message pairs first. If trimming would remove important earlier context, replace trimmed turns with a local deterministic summary in a synthetic developer message. Phase 2 must not call the LLM only to summarize or title; summaries can start as local truncation and be upgraded later behind tests.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### 4b.1 Structured Outputs with Pydantic

Phase 2's user-visible assistant answer is plain Markdown text, not JSON. Use Pydantic structured outputs only for internal metadata checks where schema validity matters, such as classifying whether an assistant response is renderable, retryable, or policy-blocked. Keep this behind the same local adapter boundary.

```python
from typing import Literal

from openai import AsyncOpenAI
from pydantic import BaseModel, Field, ValidationError


class AssistantResponseCheck(BaseModel):
    render_mode: Literal["markdown"]
    retryable: bool = Field(description="True only when the response should be regenerated.")
    user_safe_summary: str = Field(max_length=160)


async def validate_response_metadata(
    client: AsyncOpenAI,
    *,
    model: str,
    assistant_text: str,
) -> AssistantResponseCheck:
    completion = await client.chat.completions.parse(
        model=model,
        messages=[
            {
                "role": "system",
                "content": "Return metadata only. Do not quote secrets or full user content.",
            },
            {"role": "user", "content": assistant_text[:4000]},
        ],
        response_format=AssistantResponseCheck,
        temperature=0,
        max_completion_tokens=120,
    )
    parsed = completion.choices[0].message.parsed
    if parsed is None:
        raise ValueError("structured output parsing returned no parsed object")
    return parsed
```

Retry logic: allow at most one structured-output retry after `ValidationError`, `LengthFinishReasonError`, `ContentFilterFinishReasonError`, or an empty `parsed` result. Log local correlation ID, model, schema name, retry count, finish reason, and provider request ID. Surface a generic retryable provider error only if validation still fails; never show schema/debug details to the user.

Do not run this metadata check on every Phase 2 message unless it has a clear product use. If used, keep it out of the critical path or budget it explicitly, because it is a second model call.

### 4b.2 Async-First Design

FastAPI routes are async, so the adapter should use `AsyncOpenAI` and `await client.chat.completions.create(...)`. Instantiate the SDK client once through dependency injection or application startup, not per message, so connection pooling and timeout configuration remain consistent.

The common mistake is calling `asyncio.run()` from inside a FastAPI request handler. `asyncio.run()` is only for a standalone CLI entry point; inside the server, await the adapter method directly.

Phase 2 should await the complete non-streaming response. Streaming improves perceived UX but complicates disconnect handling, partial persistence, gateway buffering, and retry semantics, so it remains deferred. Await structured outputs when validation is required; do not try to validate partial streamed text as final JSON.

### 4b.3 Prompt Engineering Discipline

Keep prompt layers separate:

- Developer/system message: product identity, safety boundary, no-tool/no-cross-tenant claims, Markdown output expectations.
- User messages: exact owner-checked conversation content only.
- Assistant messages: previous persisted assistant content only.

Do not concatenate user text into the developer/system instruction. Do not include secrets, bearer tokens, cookies, provider keys, CSRF tokens, raw error dumps, or hidden authorization metadata in model context.

Few-shot examples are not needed for Phase 2 normal chat. If later needed for style consistency, put short static examples in the developer message so prompt caching can reuse the prefix. Dynamic few-shot retrieval, RAG examples, and external knowledge snippets are out of scope.

Set output limits explicitly on every call. Use `max_completion_tokens=800` for the initial prototype and make higher limits a product decision. Do not leave output unbounded in production.

### 4b.4 Context Window Management

For conversational chat, use local summarization/truncation rather than provider conversation state. Select the newest messages that fit the budget, preserve complete user/assistant pairs where possible, and never include another user's messages. If a thread exceeds the budget:

1. Keep the first developer/system prompt unchanged.
2. Keep the latest user turn.
3. Keep as many recent complete turns as possible.
4. Replace older turns with a local deterministic summary or "older context omitted" marker until a tested summarizer is introduced.

RAG, reranking, web search, and tool context are not part of Phase 2.

### 4b.5 Cost and Latency Budget

Use a per-call budget formula because OpenAI-compatible providers can have different pricing:

```text
estimated_cost =
  (prompt_tokens / 1_000_000 * input_price_per_1m) +
  (completion_tokens / 1_000_000 * output_price_per_1m)
```

OpenAI's June 11, 2026 public pricing lists `gpt-5.4-mini` at `$0.75` input and `$4.50` output per 1M tokens on standard short-context pricing. At 2,000 input tokens and 500 output tokens, that is roughly `$0.00375` per response before cached-input discounts. If the configured provider or model differs, put its prices in environment-backed settings and log estimated token cost from SDK usage fields.

Cache only where it is safe:

- Exact-match response caching is acceptable only for anonymous/static prompts, not private user conversations by default.
- Semantic caching is not appropriate for private direct chat in Phase 2 because it risks cross-user leakage and surprising reuse.
- Prompt caching benefits come mostly from a stable developer/system prefix; keep static instructions first and variable conversation content later.

Cheaper sub-task routing is limited in this phase. Use the same configured chat model for the user response. If adding internal classifiers later, route them through a separate adapter method, a cheaper configured model, strict Pydantic output, and independent token/cost logging.

## 5. Evaluation Strategy

Phase 2 evals answer: "Can an authenticated user hold a durable, private direct-LLM conversation without ownership, retry, failure, rendering, or secret-handling regressions?" Code-based checks are the release gate for deterministic security, idempotency, persistence, and rendering behavior. LLM judges are limited to subjective response quality and instruction-following checks, and only after calibration against human labels. Human review is used for AppSec/privacy calibration and ambiguous UX cases.

Out of scope for this eval plan: Google Search grounding, Python sandbox behavior, RAG metrics, arbitrary tools, streaming, organization/shared workspaces, and provider-side conversation state.

### Critical Failure Modes Confirmed

1. Cross-user access or inference of another user's conversations or messages.
2. Duplicate submit/retry creates duplicate provider calls, duplicate user messages, or out-of-order assistant messages.
3. Provider failure is hidden by fabricated successful assistant content or an error without a stable correlation ID.
4. User/model Markdown executes raw HTML, script handlers, unsafe URLs, or code in the browser.
5. Secrets, bearer tokens, cookies, provider keys, or sensitive raw content appear in logs, telemetry, model context, or user-visible errors.

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| Private conversation boundary | PASS: Every create/list/retrieve/delete/send/retry/history-context path constrains `conversation_id` or `message_id` with authenticated `user_id` and required `chat:*` scope in the same data-access path; a second user gets denied/not-found behavior with no title, message count, timing clue, provider call, or audit content from the target conversation.<br>FAIL: A route fetches by resource ID alone, allows append/delete/read across users, leaks existence metadata, or includes another user's turn in model context.<br>Measurement: Code-based two-user integration/security tests with DB assertions, provider-call spy assertions, and AppSec review of query patterns. | Code + Human | Critical |

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Auth, scope, and ownership gate | Any chat read/mutation/send/retry/delete request | Block with authenticated/forbidden/not-found response as appropriate; enforce `chat:read`/`chat:write` and owner constraint before DB content return or provider call; emit redacted event with correlation ID. |
| Provider-call deny check | Request fails auth/scope/ownership, conversation is soft-deleted, or conversation is not owned | Block provider invocation entirely; tests must prove fake provider call count remains zero. |
| Idempotency and in-flight turn gate | Duplicate `client_message_id`, browser retry, explicit retry, or concurrent pending assistant turn in one conversation | Reuse existing accepted user message or return stable pending/conflict state; never create duplicate provider work or out-of-order assistant rows. |
| Provider failure normalizer | OpenAI-compatible adapter timeout, rate limit, auth/config error, empty response, or 5xx | Persist/update failed assistant placeholder or return retryable inline error with correlation ID; never persist fabricated successful assistant content. |
| Secret and raw-content redaction | Any log, trace, audit metadata, error response, or provider exception path | Redact/drop credentials, cookies, API keys, bearer tokens, CSRF values, full prompts, and full assistant content before emission; fail CI on canary-secret leakage. |
| Markdown/code sanitizer | Rendering any user or assistant message in history or fresh response | Render raw HTML as escaped/inert text, allow only safe link schemes, isolate external links, and keep fenced code non-executable. |
| Phase 2 no-tool boundary | Any code path attempts to pass `tools`, `tool_choice`, web search, file search, code interpreter, MCP config, or arbitrary tool metadata to the provider | Reject during adapter validation or fail tests; keep direct chat provider payload tool-free. |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix with OpenTelemetry spans around FastAPI chat routes and the local OpenAI-compatible chat adapter. Store sanitized metadata only: correlation ID, hashed user/conversation/message identifiers, model, provider request ID, latency, timeout/retry count, token counts, estimated cost, status, retryability, and sanitized error code. Do not export raw prompts, raw assistant text, bearer tokens, cookies, API keys, CSRF values, or provider exception bodies.

**Key Metrics to Track:**

| Metric | Alert Threshold | Notes |
|--------|-----------------|-------|
| Cross-user authorization failure safety | Any successful cross-user read/write/delete/send, or any provider call after an auth/scope/ownership denial | Critical page/blocker; expected rate is exactly zero. |
| Secret/raw-content leakage | Any canary secret, token, cookie, API key, full prompt, or full assistant body in logs/traces/errors | Critical page/blocker; rotate affected credentials if real secrets were exposed. |
| Duplicate side effects per `client_message_id` | Any reference-dataset failure, or >0.1% duplicate persisted user/assistant rows or duplicate provider calls in sampled production/demo traffic | High severity; investigate transaction/idempotency path. |
| Provider failure rate and retry health | Provider error rate >5% for 15 minutes, failed placeholder ratio >10%, or retry success rate drops below 70% for retryable failures | High severity; check provider status, config, timeout, and user-facing failure clarity. |
| Chat latency and timeout drift | p95 route latency >20s for 15 minutes, p99 >30s, or timeout rate >2% | High severity for UX and cost; inspect provider latency, context size, and retries. |
| Token/cost budget drift | Median prompt tokens or estimated cost per successful turn exceeds 2x the baseline for a day, or any request exceeds configured max context budget | Medium severity; review context trimming and message selection. |
| Rendering/security regression | Any automated sanitizer/browser test failure, unsafe link rendered as clickable, or script/event execution observed | Critical release blocker. |

**Alert Thresholds:**

Critical alerts page/block release immediately for BOLA success, provider call after denied ownership, secret/raw-content leakage, executable Markdown payloads, or fabricated successful assistant content after provider failure. High alerts require same-day triage for duplicate side effects, provider error spikes, correlation-ID loss on failures, and sustained latency/timeout drift. Medium alerts feed backlog work for tone-quality drift, token/cost growth, or confusing but non-dangerous failure copy.

**Smart Sampling Strategy:**

Capture 100% sanitized traces for provider failures, retry attempts, concurrency conflicts, authorization denials, delete operations, sanitizer-blocked content, high-token requests, slow requests, and any redaction hit. Sample 5-10% of successful normal chat traces as metadata-only observability. For human review, select at least 20 interactions per release candidate from the reference dataset and high-signal runtime samples; use sanitized/exported content only when needed for calibration and never include other users' data or secrets. Oversample LLM-judge disagreements, off-scope prompts, long conversations, and cases where user retries or abandons after a failure row.
