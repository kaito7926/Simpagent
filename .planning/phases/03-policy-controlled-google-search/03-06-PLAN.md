<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Admin orchestration API -> runtime settings storage | Privileged user input becomes persisted provider-selection state |
| Runtime settings storage -> provider resolver | Persisted override values influence readiness and live search execution |
| Admin write action -> evidence store | Security-sensitive orchestration changes must be auditable and correlated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-03-22 | Elevation | provider-override write path | mitigate | Restrict D-12 write operations to existing admin RBAC and fail closed on under-scoped or non-admin principals |
| T-03-23 | Tampering | persisted override value | mitigate | Accept only `gemini`, `firecrawl`, or cleared null in repository/service validation and reject all other strings |
| T-03-24 | Repudiation | orchestration changes | mitigate | Record correlated admin/security evidence for override set and clear operations in `admin_evidence.py` |
| T-03-25 | Spoofing | readiness vs runtime provider | mitigate | Use one effective-provider resolver in `provider_status.py` and `chat_turns.py` so admin reads and live execution cannot diverge |
| T-03-26 | Denial | stale override state | mitigate | Explicit clear semantics and capability-check tests ensure the runtime returns to the environment default after override removal |
</threat_model>
