# Quick Task 260614-s42: Explicit tool-mode chat API

## Goal

Allow an authenticated user to explicitly request Web Search Agent or Python Sandbox Agent from the normal chat API, then document the request contract for a future UI agent that will add two checkbox controls.

## Must Haves

- Existing chat message endpoints keep their current default behavior when no mode is supplied.
- `tool_mode: "google_search"` forces a bounded Web Search Agent turn even when the prompt text would not be auto-routed to search.
- `tool_mode: "python"` forces a bounded Python Sandbox Agent turn even when the prompt text would not be auto-routed to Python.
- Tool authorization stays backend-owned: missing `tool:websearch` or `tool:python` returns a denied tool result and does not call the worker/planner.
- Python execution continues through the existing sandbox coordinator path; no backend or host Python execution path is introduced.
- Vietnamese documentation tells a UI-building AI how to wire two mutually exclusive checkbox/toggle controls.

## Tasks

1. Extend backend chat request schemas and coordinator routing to accept optional explicit tool mode.
2. Add integration tests for forced Google Search, forced Python, missing Python permission, and invalid mode rejection.
3. Update Vietnamese documentation with request examples, response metadata expectations, and UI control notes.
4. Run focused backend tests for the changed contracts.
